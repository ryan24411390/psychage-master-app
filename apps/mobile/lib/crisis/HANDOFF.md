# Crisis data layer ‚Äî handoff to the crisis-UI wave

This `lib/crisis/` module is the **CT3 data layer** delivered by the work order
"CT3 Crisis Dataset ‚Äî Supabase persistence + per-client data layer." It is
**standalone and not yet wired into the existing S11/S12 screens.** This note
explains what exists, why it isn't wired, and what the crisis-UI wave must do to
adopt it.

## What this module is

- `types.ts` ‚Äî the public contract: `getCrisisResources(iso2)` /
  `listCrisisCountries()` returning a **6-field** `HelplineRow`
  `{ name, description, callNumber, textCapable, textNumber, region }`.
- `store.ts` ‚Äî offline-first resolution (MMKV cache ‚Üí silent network refresh ‚Üí
  committed bundle floor), DI-tested.
- `index.ts` ‚Äî real wiring (storage adapter + anon Supabase client + bundle).
- `../../data/crisis/crisis-bundle.json` (+ `.types.ts`) ‚Äî the committed,
  verified-only offline floor, generated by `scripts/export-crisis-bundle.ts`.

It reads fully offline; the network is never required (rules/offline.md ¬ß6).

## Why it is NOT wired into S11/S12 yet

The existing crisis UI (`features/crisis/`) was built by an earlier wave against
a **frozen, intentionally different** presentational schema
(`features/crisis/helpline-schema.ts`):

| | This data layer (CT3 / web-parity) | Frozen `features/crisis` schema |
|---|---|---|
| desc field | `description` | `fiveWordDesc` |
| text number | `textNumber` (separate) | ‚Äî (none; texting reuses `callNumber`) |
| call number | `callNumber: string \| null` | `callNumber: string` (**required, non-null**) |
| shape | flat rows + `getCrisisResources(iso2)` | region-keyed `CrisisDataset` maps |
| data | real CT3 verified rows (Supabase) | dummy fixture (`helplines.fixtures.ts`) |

The two are **not interchangeable**, and bridging them loses safety-relevant
data:

- **Text-only lines cannot be represented in the frozen schema.** Real verified
  rows exist with `callNumber: null` + a distinct `textNumber` (e.g. US **Crisis
  Text Line ‚Äî text 741741**, GB Shout 85258, IE 50808). The frozen 5-field row
  requires a non-null `callNumber` and has no `textNumber`, so these lines would
  be dropped or corrupted.
- **`emergency_note`** (e.g. "110 police, 120 ambulance") has no home in the
  frozen `emergencyByRegion` flat map.
- The CT3 roster is **69 countries (>20)**, which per the fixture's own note
  requires FlashList in `RegionPickerView` (currently FlatList for ‚â§20).

These are schema/UI decisions owned by the crisis-UI wave (and clinically by
Dr. Dobson), not something this data-layer PR should unilaterally change. Per the
work-order constraint, this PR does **not** touch any crisis UI/fixture.

## What the crisis-UI wave must do to adopt this

1. Decide the reconciliation: either (a) extend the frozen `HelplineRow` (add
   `textNumber`, make `callNumber` nullable, rename `fiveWordDesc`‚Üí`description`
   or map between them) ‚Äî requires unfreezing with sign-off ‚Äî or (b) keep the
   presentational schema and map at the boundary, accepting the documented data
   loss (NOT recommended for text-only crisis lines).
2. Replace `features/crisis/helplines.fixtures.ts` (`CRISIS_DATASET`) ‚Äî its
   header says "Replace this whole module wholesale when CT3 lands" ‚Äî by adapting
   from `getCrisisResources` / `listCrisisCountries` (or directly from the
   committed bundle).
3. Handle the >20-country roster (FlashList) in `RegionPickerView`.
4. Wire `refreshCrisisCache()` to app-foreground (rules/offline.md ¬ß6: once per
   day on foreground).

## Operational

- **Regenerate the bundle** when the dataset changes:
  `SUPABASE_URL=‚Ä¶ SUPABASE_SERVICE_ROLE_KEY=‚Ä¶ pnpm --filter @psychage/mobile export:crisis`
  (service-role key from the environment ‚Äî never committed).
- **Runtime env:** set `EXPO_PUBLIC_SUPABASE_URL` and
  `EXPO_PUBLIC_SUPABASE_ANON_KEY` (anon key is public; RLS is the gate). Without
  them, the client is `null` and reads serve the committed bundle ‚Äî crisis still
  works fully offline.
